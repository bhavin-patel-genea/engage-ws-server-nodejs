```
title Engage Gateway Client Mode - Full Flow

participant "Genea Cloud\n(WebSocket Service)" as Genea
participant "Allegion Engage\nGateway" as GW
participant "Lock\n(Edge Device)" as Lock

note over Genea,Lock: LEGEND\nSolid arrow  (->)  = request / command\nDashed arrow (-->) = response / event\nAll WebSocket messages are JSON wrapped in ENGAGE sub-protocol.\nHTTP calls in Stages 1-2 are plain HTTP/HTTPS — not WebSocket.

note over Genea: PRE-REQUISITE: One-Time Setup by Genea (done once, not per customer)\n\n1. Generate self-signed certificates via OpenSSL\n   ✅ Self-signed confirmed by Allegion (#651044)\n   (see GENEA CERTIFICATE STRATEGY section at end for full details)\n\n2. Register each customer site via Allegion SAM API\n   POST https://partner.lockwebserv.com/engage/sites/register\n   Body: { SiteName, Base64Key, IsEngageManaged, SiteSoftwareKey }\n   Response: unique Site Key (32-byte AES-256) per customer site\n\n3. Store in Genea:\n   - rootca.der          generated by Genea via OpenSSL (not from Allegion)\n                         shared across ALL customers ✅ Confirmed (#651044)\n   - server.crt + key    generated by Genea, signed by rootca.key\n                         shared across ALL customers ✅ Confirmed (#651044)\n   - Site Key            UNIQUE per customer site (from Allegion SAM API)\n   - Gateway SN list     which SNs belong to which site\nTLS: TLS 1.2 / TLS 1.3 (confirmed by Allegion 2026)\n\nSECURITY NOTE — AES-256-CBC with Zero IV:\n   Used for credential encryption in Stage 2.\n   Observed in Allegion POC, not explicitly documented in App Notes.\n   Anti-replay: +/-300s timestamp window.\n\nENDPOINTS:\n   CA bootstrap : http://host:80/engage/newCA/current  (plain HTTP — mandatory)\n   Credentials  : https://host:443/engage/newCredentials\n   WebSocket    : wss://host:443/engage_wss

note over Genea,Lock: STAGE 0: Gateway Provisioning (One-Time, Before First Connection)\nBefore a gateway can connect, Genea must:\n1. Register the gateway Serial Number (SN) to a customer site in Genea DB\n2. Provision the correct Site Key to that site's record\n3. Configure the gateway via ENGAGE Mobile App:\n   - Server URL    : wss://host:443/engage_wss\n   - CA Server URL : http://host:80/engage/newCA/current\n   - Keep Alive    : 30s\n   - IP Mode       : DHCP\n4. Perform FDR (Factory Default Reset) on the gateway to apply config\nWithout SN registration in Genea DB, Stage 1 CA lookup will fail\nbecause Genea cannot determine which Site Key to use for HMAC.

note over GW,Genea: STAGE 1: Certificate Setup\nPurpose: Gateway must trust Genea's server certificate before any TLS connection.\n\nTRIGGERS (Root Cert App Note v1.2 p4):\n  1. First commissioning (no cert installed yet)\n  2. TLS certificate validation error during any HTTPS connection\n     (automatic — firmware detects the error and begins Stage 1)\n  3. Manual: ENGAGE Mobile App -> "Update from Server"\n  4. Factory Default Reset (FDR)\nThe Root Cert App Note does NOT list cert expiry as a separate trigger;\nexpiry would cause a TLS validation error, which triggers case (2).\n\nWHY PLAIN HTTP FOR CA DOWNLOAD?\nBootstrap deadlock: the gateway needs rootca.der to trust TLS,\nbut cannot use TLS to download it yet — chicken-and-egg problem.\nPlain HTTP on port 80 is Allegion protocol design — not a workaround.\nIntegrity is guaranteed by HMAC-SHA1 per RFC 2104, not by transport encryption.\nSource: WebSockets App Note v1.06 p8, Root Cert App Note v1.2 p5\n\nWHY DER FORMAT (not PEM)?\nDER = raw binary encoding of the X.509 certificate.\nPEM = same data but Base64-wrapped with -----BEGIN CERTIFICATE----- header.\nGateway firmware expects raw binary DER — it cannot parse text-wrapped PEM.\n"This certificate should be served as a DER encoded certificate file (binary)"\nSource: Root Cert App Note v1.2 p5-6

note over GW,Genea: STAGE 1a — Gateway requests cert URL + integrity promise

GW->Genea: GET http://host:80/engage/newCA/current\n?serialNumber=<Base64(SN)>&hashType=primary&v=2

note left of Genea: Genea CA Server (plain HTTP, port 80):\n1. Parse serialNumber from query string\n   (16-byte hex SN in Base64 encoding — Root Cert App Note v1.2 p5)\n2. Look up Site Key for this SN in Genea DB\n   [FAIL: SN not found -> 404, gateway retries]\n3. Read rootca.der bytes from disk (same file for ALL customers)\n4. Compute HMAC-SHA1(rootca.der bytes, SiteKey) -> Base64 encode\n   Algorithm: HMAC-SHA1 per RFC 2104 (Root Cert App Note v1.2 p5)\n   This is a PROMISE: "the file at cert_url will produce this fingerprint"\n   Same cert file + different SiteKey = different hash per customer:\n     Customer A (Key AAA): hash = Base64(HMAC-SHA1(rootca, AAA))\n     Customer B (Key BBB): hash = Base64(HMAC-SHA1(rootca, BBB))\n   An attacker can see the hash but cannot forge one without the SiteKey.\n5. Build cert_url pointing to /engage/certificates endpoint\n   cert_url max length: 256 chars for GWE client mode\n   (Root Cert App Note v1.2 p5)\n\nhashType param: "primary" = verify using SiteKey.\n  If omitted, "primary" is implied (Root Cert App Note v1.2 p5).\nv=2: API version, required by firmware.

Genea-->GW: 200 OK\n{"cert_url":"http://host:80/engage/certificates",\n "hash":"<Base64(HMAC-SHA1(rootca.der, SiteKey))>"}\n\nGateway now holds:\n  promise  = hash value above\n  cert_url = where to download the cert

note over GW,Genea: STAGE 1b — Gateway downloads cert and verifies the promise

GW->Genea: GET http://host:80/engage/certificates\n(URL taken from cert_url field in Stage 1a response)

note left of Genea: Serves raw rootca.der binary — no modification, no HMAC embedded.\nContent-Type: application/octet-stream (required — Root Cert App Note v1.2 p6)\nContent-Length: included (required for CTE, LE, LEB, NDEB devices)\nTransfer-Encoding: omit (safest for all device types)\nSame binary for every gateway on every customer site.\nSecurity comes from the HMAC check the gateway performs, not from this response.

Genea-->GW: rootca.der (raw binary, DER format, same for all customers)

note right of GW: Gateway verifies the promise made in Stage 1a:\n\n  recompute = HMAC-SHA1(downloaded bytes, own SiteKey)\n\n  recompute == promise (hash from Stage 1a)?\n  YES — bytes are genuine, nobody swapped the file in transit\n           Gateway installs cert as its trusted CA (trust anchor)\n           All future TLS connections to Genea validated against this cert\n  NO  — bytes were tampered in transit (possible MITM attack)\n           Gateway discards the cert, aborts Stage 1, retries later\n\nADDITIONAL VALIDATION (Root Cert App Note v1.2 p6):\n  "If the provided location is invalid, the format of the new\n  certificate is not correct, or the device is unable to validate\n  the new certificate successfully against the hash value provided,\n  the process is terminated and the device will not update its\n  internal certificate."\n\nWHY THIS IS SECURE OVER PLAIN HTTP:\n  Attacker can intercept and READ the cert_url and hash — that is fine.\n  Attacker CANNOT forge a valid hash for a fake cert without the SiteKey.\n  SiteKey never travels on the wire — attacker never has it.\n  Swapping the cert = hash mismatch = gateway rejects it.

note over GW,Genea: Stage 1 Result: Gateway now trusts Genea Root CA.\nStage 1 is NOT repeated on 24h reconnect — CA is cached on the gateway.\nStage 1 repeats automatically on TLS certificate validation error\n(e.g., cert expired, cert rotated, HMAC mismatch, or FDR).\nCan also be triggered manually via ENGAGE Mobile App "Update from Server".\nSource: Root Cert App Note v1.2 p4

note over GW,Genea: STAGE 2: Credential Exchange\nGateway requests a one-time password before WebSocket upgrade.\nUses HTTPS (TLS validated via rootca.der installed in Stage 1).

GW->Genea: POST https://host:443/engage/newCredentials\nBody: Base64( SN : hexEncode(AES-256-CBC(payload, SiteKey, IV=0)) : 1 )\n\nEncrypted payload layout (32 bytes total):\n  Bytes  0-7  : zero-padded\n  Bytes  8-15 : Unix epoch timestamp (uint64, little-endian)\n  Bytes 16-31 : Serial Number (16 raw binary bytes)\n\nSecurityRevision "1" is Allegion-defined.\nZero IV is observed in Allegion POC implementation\n(not explicitly documented in App Note v1.06).

note left of Genea: Genea validates (all checks must pass):\n1. Decode Base64 body\n2. Split on ":" -> [plainSN, hexPayload, secRev]\n3. Decode hex payload -> 32 bytes\n4. Decrypt: AES-256-CBC(payload, SiteKey, IV=zeros)\n5. Extract timestamp from bytes 8-15\n6. Verify timestamp is within +/-300s of server time (anti-replay)\n   [FAIL: drift > 300s -> 400, gateway must sync clock and retry]\n7. Extract SN from bytes 16-31 as raw hex string\n8. Verify decryptedSN == plainSN (case-insensitive)\n   [FAIL: mismatch -> 415, possible wrong Site Key]\n9. Generate 32-byte cryptographically random password\n10. Store: SN -> tempPassword (in-memory, single-use)\n11. Credential is valid for next WebSocket upgrade only\n\nSECURITY NOTE: Timestamp endianness (LE vs BE) is ambiguous\nin App Note v1.06. Server tries both and picks whichever gives\nsmaller drift. This is a known Allegion protocol ambiguity\n(not clarified in either App Note).

Genea-->GW: 200 OK\nBody: <32-byte random password, Base64-encoded>

note over GW,Genea: STAGE 3: WebSocket Upgrade\nThe gateway switches from normal HTTP (request → response → done) to a persistent WebSocket connection (always open, both sides can send anytime).

GW->Genea: HTTP Upgrade -> WSS (TLS 1.2 or 1.3)\nGET /engage_wss HTTP/1.1\nAuthorization: Basic Base64(SN:TempPassword)\nUpgrade: websocket\nSec-WebSocket-Protocol: engage.v1.gateway.allegion.com\nSec-WebSocket-Version: 13\n\nTLS 1.2 confirmed in practice (Root Cert App Note v1.2 p17).\nTLS 1.3 confirmed by Allegion team (2026).

note left of Genea: Genea WebSocket upgrade validation:\n1. Extract SN and TempPassword from Basic Auth header\n2. Look up stored SN -> TempPassword mapping\n3. Match   -> accept upgrade, DELETE stored credential (single-use)\n4. Mismatch -> 401 Unauthorized (gateway retries from Stage 2)\n   NOTE: 403 is intentionally avoided — some gateway firmware\n   versions stop retrying on 403 but do retry on 401.\n5. Check Sec-WebSocket-Protocol header:\n   Accepts: engage.v1.gateway.allegion.com\n            engage.v1.edgedevice.allegion.com\n   Mismatch -> 400 Bad Request\n6. If SN already has an active connection:\n   Close stale socket first, then accept new connection.\n   (Handles gateway restart or network blip mid-reconnect)\n7. Register connection: validConnections[SN] = socket\n8. Start TCP keep-alive probe (30s OS-level)\n9. Start WebSocket ping/pong heartbeat (see Stage 4 note)

Genea-->GW: 101 Switching Protocols\nSec-WebSocket-Protocol: engage.v1.gateway.allegion.com\n101 = "switching from HTTP to WebSocket now." Sub-protocol = both sides agree to speak the same ENGAGE message format before data flows.

note over GW,Genea: STAGE 4: Event Subscription + Heartbeat Setup\nServer sends subscription immediately after upgrade.\nNo subscription = gateway sends NO events (Allegion protocol requirement).\nSubscription is stateless on the gateway — must be re-sent after every reconnect.

Genea->GW: EngageEventSubscription\n{"subscriptionId":1,"subscription":[\n  {"source":"gateway","eventingEnabled":true,"subscriptionBody":{}},\n  {"source":"edgeDevice","eventingEnabled":true,"subscriptionBody":{}}]}\n\nNote: No ACK is expected. Server proceeds regardless.\nIf subscription message is lost, gateway will not send events\nand the connection must be re-established.

note over GW,Genea: HEARTBEAT (runs continuously from Stage 4 until disconnect)\nAllegion says WebSocket ping/pong is optional for the server and should be tuned carefully because some devices may be busy long enough to miss heartbeat responses.\nFor this POC, Genea uses a production-oriented default of one WebSocket PING every 15 seconds and disconnects after 3 missed PONGs (about 45 seconds total).\nIf 3 consecutive PINGs have no PONG response:\n  -> Server closes the socket with code 1001 (Going Away)\n  -> _connectionLost fires, gateway SN removed from validConnections\n  -> Gateway must restart from Stage 2 (not Stage 1)\n\nKeep-Alive header in Mobile App (30s) is a TCP-level setting\nseparate from the WebSocket ping/pong (15s) above.

note over Genea,Lock: STAGE 5: Device Discovery (Server-Initiated, After Stage 4)

Genea->GW: EngageRequest GET /edgeDevices/linkList\n{"requestId":1,"request":{"method":"GET","path":"/edgeDevices/linkList","messageBody":""}}

GW-->Genea: EngageResponse 200\n{"responseStatus":"200","messageBody":"{\\"edgeDeviceLinkList\\":[{\\"linkId\\":\\"dev001\\",\\"deviceName\\":\\"Front Door\\",\\"modelType\\":\\"...\\"}, ...]}"}\n\n[TIMEOUT: if no response in 15s -> log error, skip lock status enrichment]

Genea->GW: EngageRequest GET /edgeDevices/lockStatus\n{"requestId":2,"request":{"method":"GET","path":"/edgeDevices/lockStatus","messageBody":""}}

GW-->Genea: EngageResponse 200\n{"responseStatus":"200","messageBody":"{\\"edgeDeviceLockStatus\\":[{\\"linkId\\":\\"dev001\\",\\"lockState\\":\\"secure\\"}, ...]}"}\n\n[OPTIONAL: if this call fails, device list is still shown with lockState="?"]

note over Lock,Genea: STAGE 6: Credential Provisioning (Access Policy Push)

Genea->GW: EngageRequest PUT /edgeDevices/{linkId}/database\n{"requestId":3,"request":{"method":"PUT","path":"/edgeDevices/dev001/database","messageBody":"<credential DB>"}}\n\nThe database contains: badge credentials, access schedules, PIN codes.\nMax BLE transfer rate to lock: 768 bytes/sec.\nThis determines whether the lock can make LOCAL access decisions (offline mode).

GW->Lock: Database push over BLE\n(lock does not need to be online to Genea during badge reads)
Lock-->GW: Database processed
GW-->Genea: EngageResponse 202 + audit confirmation\n\n[If lock is unreachable over BLE: GW returns non-200 status]

note over Lock,Genea: STAGE 7: Badge Read (Access Event — Lock-Initiated)

Lock->Lock: Cardholder presents badge.\nLock makes access decision LOCALLY\nusing cached credential database (offline mode).\nNo real-time Genea query is made for the decision.
Lock-->GW: Audit event over BLE (access granted or denied)
GW-->Genea: EngageEvent push via WebSocket\n{"eventId":"...","eventType":"...","eventSource":"edgeDevice",\n "eventDeviceId":"dev001","eventBody":"{\\"badgeId\\":\\"...\\",\\"accessResult\\":\\"granted\\", ...}"}\n\nNote: eventBody is an opaque JSON string — structure varies by event type.\nGenea must parse eventBody to extract badgeId, accessResult, timestamp, etc.

note over Lock,Genea: STAGE 8: Remote Lock Control (Server-Initiated)

Genea->GW: EngageRequest PUT /edgeDevices/{linkId}/lockControl\n{"requestId":6,"request":{"method":"PUT",\n "path":"/edgeDevices/dev001/lockControl",\n "messageBody":"{\\"lockControl\\":{\\"lockState\\":{\\"nextLockState\\":\\"passage\\"}}}"}\n\nValid nextLockState values:\n  passage        = unlock (hold open)\n  secure         = lock\n  momentaryUnlock = brief unlock pulse (Quick Grant)\n  doNotDisturb   = do not disturb mode

GW->Lock: Lock control command over BLE

note right of GW: If BLE link to lock is unavailable:\n  GW returns non-200 status immediately.\nIf BLE link is available but lock rejects command:\n  GW returns error status from lock.\nTimeout: server waits 15s for GW response.\n  If no response in 15s -> 504 Gateway Timeout to caller.

Lock-->GW: Status acknowledgement over BLE
GW-->Genea: EngageResponse\n{"responseStatus":"200","messageBody":"..."}\n\n[Non-200: log error, alert Genea ops, do not update local lock state cache]

note over GW,Genea: STAGE 9: 24-Hour Reconnection Cycle\nGateway firmware enforces a maximum 24-hour connection lifetime.\nRepeats Stages 2, 3, 4 only — Stage 1 is skipped (CA already trusted).\nServer does NOT enforce the 24h limit — it is purely gateway-side.

GW->GW: Firmware timer fires (24h elapsed).\nGateway closes WebSocket connection.\nServer detects close, fires _connectionLost,\ndeletes validConnections[SN] and stored credential.

GW->Genea: POST https://host:443/engage/newCredentials\n(same as Stage 2 — full credential exchange)\n\n[FAIL path: if server is unreachable or returns error\n  -> gateway applies exponential backoff and retries\n  -> no events or commands possible until reconnected]

Genea-->GW: 200 OK TempPassword (new, single-use)
GW->Genea: WSS Upgrade (same as Stage 3)
Genea-->GW: 101 Switching Protocols
Genea->GW: EngageEventSubscription (same as Stage 4 — mandatory re-send)

note over GW,Genea: UNEXPECTED DISCONNECT HANDLING\nIf WebSocket drops unexpectedly (network loss, server restart, ping timeout):\n  -> Server: _connectionLost fires, credential and connection purged\n  -> Gateway: detects close, restarts from Stage 2 immediately\n     (does NOT wait 24h — reconnects as fast as firmware allows)\nThis means Stage 2-4 can repeat multiple times within a 24h window.


note over Genea: GENEA CERTIFICATE STRATEGY\n\nRECOMMENDATION: SELF-SIGNED CERTIFICATE (not GoDaddy / AWS ACM)\n  ✅ CONFIRMED BY ALLEGION (Dave Chisholm, #651044, 2026-03-28)\n  "Self-Signed is intended."\n\n  The gateway does NOT have a built-in public CA trust store.\n  It trusts ONLY the rootca.der downloaded via Stage 1.\n  A GoDaddy or ACM cert has ZERO trust advantage on the gateway.\n  Allegion's own production certs are self-signed (30-year validity).\n  Security comes from HMAC-SHA1 + per-customer Site Key, not cert issuer.\n\n  Why NOT GoDaddy / ACM:\n    - Gateway ignores who issued the cert — no security benefit\n    - Public CA root certs often > 2048-bit RSA or use ECDSA\n      — may VIOLATE firmware constraints (RSA max 2048-bit only)\n    - AWS ACM private keys cannot be exported — unusable directly\n    - Cert chain may exceed 4096-byte firmware limit (3 certs vs 2)\n    - Annual cost and third-party dependency with zero benefit\n    - Gateway cannot perform CRL/OCSP revocation checks\n\nWHY ONE CERTIFICATE FOR ALL CUSTOMERS (not per-customer certs):\n  ✅ CONFIRMED BY ALLEGION (Dave Chisholm, #651044, 2026-03-28)\n  "I think you can use the same CA in this case."\n  Allegion confirmed no strict policy requiring per-customer CAs.\n  Site Key provides sufficient per-customer separation.\n\n  rootca.der is the SAME file for every customer site.\n  Per-customer isolation comes from the Site Key, not the certificate:\n\n    Customer A (SiteKey AAA): HMAC = HMAC-SHA1(rootca.der, AAA)\n    Customer B (SiteKey BBB): HMAC = HMAC-SHA1(rootca.der, BBB)\n\n  Same cert file + different Site Key = different HMAC per customer.\n  A gateway from Customer A cannot use Customer B's HMAC to verify\n  the cert — the hash won't match. No need for separate certs.\n\n  Benefits of shared cert:\n    - One cert to manage, one rotation schedule\n    - No per-customer CA generation or storage\n    - Simpler ops at scale (hundreds of sites, one rootca.der)\n\nCERTIFICATE FILES — WHAT EACH ONE DOES:\n\n  All three files are self-signed, generated via OpenSSL.\n  No ACM, no GoDaddy, no public CA involved.\n\n  rootca.key  = ROOT CA PRIVATE KEY (master key — generates everything)\n    Used to: sign rootca.crt and server.crt\n    Store securely: AWS KMS / HashiCorp Vault / HSM in production\n    Never deployed to servers — only used during cert generation\n\n  rootca.crt  = ROOT CA CERTIFICATE (PEM format — human-readable)\n    Contains the public key of rootca.key\n    Used to: sign server.crt, and as source to derive rootca.der\n    Intermediate file — not directly used by gateway or server\n\n  rootca.der  = ROOT CA CERTIFICATE (DER format — binary)\n    SAME cert as rootca.crt, just different encoding (binary vs text)\n    Gateway downloads this in Stage 1 and installs it as trust anchor\n    It answers: "Who do I trust?"\n    Served via: config.json → root_ca_file\n\n  server.key  = SERVER PRIVATE KEY\n    Generated independently (not derived from rootca)\n    Used by HTTPS/WSS server to prove ownership of server.crt\n    Never shared, never leaves the server\n    Served via: config.json → ssl_info.ssl_key\n\n  server.crt  = SERVER TLS CERTIFICATE\n    Signed by rootca.key — this is what links it to rootca.der\n    Server presents this during every TLS handshake (Stages 2, 3)\n    Gateway checks: "Was this cert signed by my trusted rootca.der?"\n      YES → TLS succeeds → proceed\n      NO  → TLS fails → re-trigger Stage 1 → infinite loop\n    Must have SAN matching server domain (e.g., DNS:mercury-api.dev-sequr.io)\n    Served via: config.json → ssl_info.ssl_cert\n\nHOW CERTIFICATES ARE DERIVED (generation chain):\n\n  Step 1: Generate Root CA (done ONCE, shared across all customers)\n    openssl genrsa -out rootca.key 2048\n    openssl req -x509 -new -nodes -key rootca.key -sha256 -days 3650 -out rootca.crt\n\n  Step 2: Export Root CA to DER format (what the gateway downloads)\n    openssl x509 -in rootca.crt -out rootca.der -outform DER\n    rootca.crt and rootca.der are the SAME certificate, different encoding:\n      PEM = Base64 text with -----BEGIN CERTIFICATE----- header\n      DER = raw binary (no header, no Base64)\n    Gateway firmware requires DER — it cannot parse PEM.\n\n  Step 3: Generate Server TLS cert (signed by Root CA)\n    openssl genrsa -out server.key 2048\n    openssl req -new -key server.key -out server.csr -subj "/CN=your-domain.com"\n    echo "subjectAltName=DNS:your-domain.com" > san.ext\n    openssl x509 -req -in server.csr -CA rootca.crt -CAkey rootca.key \\\n      -CAcreateserial -out server.crt -days 365 -extfile san.ext\n\n  The trust chain:\n    rootca.key ──generates──► rootca.crt ──exports──► rootca.der (gateway trust)\n        │\n        └──signs──► server.crt + server.key (server TLS identity)\n\n  ALL FILES MUST COME FROM THE SAME CHAIN.\n  If server.crt is from GoDaddy but rootca.der is self-signed:\n    Gateway trusts: self-signed rootca.der\n    Server shows:   GoDaddy cert\n    Result: chain mismatch → TLS fails → Stage 1 loop forever\n\n  config.json mapping:\n    "ssl_cert"     → server.crt   (self-signed, from YOUR rootca)\n    "ssl_key"      → server.key   (self-signed, from YOUR rootca)\n    "root_ca_file" → rootca.der   (DER export of YOUR rootca.crt)\n    NEVER point ssl_cert/ssl_key to public certs or other CA certs.\n\n  For customer questions about "self-signed":\n    "The ENGAGE gateway uses a private trust model — it does not\n    recognize public CAs. Our Root CA is validated via HMAC-SHA1\n    signed with your site-specific key. This is the same model\n    Allegion uses for their own certs (30-year self-signed Root CA).\n    A public CA provides no additional security for gateway connections."\n\nPRODUCTION DEPLOYMENT — TWO SEPARATE CERT CHAINS (NEVER MIX)\n\n  ACM/GoDaddy cert on ALB for browsers (port 443).\n  Self-signed cert on EC2/ECS for gateways (port 8999 + 8081).\n  They never mix.\n\n  Example using sequr.io:\n\n  DNS SETUP:\n    engage-api.sequr.io  → ALB (for browser/API traffic)\n    engage-gw.sequr.io   → EC2/ECS public IP or NLB (for gateway traffic)\n    (can be same EC2, different ports — or separate services)\n\n  TRAFFIC FLOW:\n\n    Browser / Genea API:\n      https://engage-api.sequr.io/api/locks\n      → ALB (port 443, ACM cert *.sequr.io, green padlock)\n      → Target Group → EC2:3000 (Genea API, no TLS here — ALB terminates)\n\n    Gateway Stage 1 (CA bootstrap, plain HTTP):\n      http://engage-gw.sequr.io:8081/engage/newCA/current\n      → EC2:8081 directly (no ALB, no cert — plain HTTP required)\n\n    Gateway Stage 2-3 (credentials + WebSocket, self-signed TLS):\n      wss://engage-gw.sequr.io:8999/engage_wss\n      → EC2:8999 directly (self-signed server.crt, NOT through ALB)\n\n  WHY NOT PUT GATEWAY TRAFFIC THROUGH ALB?\n    ALB would terminate TLS with ACM cert (*.sequr.io).\n    Gateway trusts rootca.der, NOT ACM.\n    Result: TLS mismatch → Stage 1 loop forever.\n    Gateway must connect directly to EC2/ECS where self-signed cert is served.\n\n  AWS SETUP:\n    ALB:\n      - Listener 443 → ACM cert (*.sequr.io) → Target Group → EC2:3000\n      - For browser dashboard + Genea REST API only\n    EC2 Security Group:\n      - Port 3000 from ALB security group (API traffic)\n      - Port 8999 from gateway IP range (WSS — self-signed TLS)\n      - Port 8081 from gateway IP range (CA bootstrap — plain HTTP)\n\n  WHAT CHANGES IN server.crt SAN:\n    When generating server.crt, SAN must match the gateway-facing domain:\n      echo "subjectAltName=DNS:engage-gw.sequr.io" > san.ext\n    NOT *.sequr.io (that is the ACM cert for ALB — different chain)\n\n  config.json FOR PRODUCTION:\n    {\n      "server_port": 8999,\n      "ssl_info": {\n        "ssl_enabled": true,\n        "ssl_cert": "./config/server.crt",\n        "ssl_key": "./config/server.key"\n      },\n      "ca_server_port": 8081,\n      "public_hostname": "engage-gw.sequr.io",\n      "root_ca_file": "./config/rootca.der",\n      "site_key_file": "./config/sitekey"\n    }\n    ssl_cert/ssl_key = ALWAYS self-signed (never ACM/GoDaddy)\n    public_hostname  = gateway-facing domain (used in cert_url response)\n\n  GATEWAY MOBILE APP CONFIGURATION:\n    Server URL   : wss://engage-gw.sequr.io:8999/engage_wss\n    CA Server URL: http://engage-gw.sequr.io:8081/engage/newCA/current\n\n  SUMMARY TABLE:\n    Traffic        | Domain                | Port | Cert            | Terminates at\n    ---------------+-----------------------+------+-----------------+--------------\n    Browser/API    | engage-api.sequr.io   | 443  | ACM *.sequr.io  | ALB\n    Gateway WSS    | engage-gw.sequr.io    | 8999 | Self-signed     | EC2/ECS\n    Gateway CA     | engage-gw.sequr.io    | 8081 | None (HTTP)     | EC2/ECS
```
